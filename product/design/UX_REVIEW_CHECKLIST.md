# UX Review Checklist

**Version**: 2.0  
**Owner**: Designer / Product Manager

---

## Purpose

Structured checklist for reviewing UX designs before approval. Use this checklist in the design review step of the design process.

---

## Product Fit

- [ ] The design solves the stated user problem (from PRD or request)
- [ ] The design aligns with user goals identified in research or personas
- [ ] The design doesn't introduce unnecessary complexity
- [ ] The feature is discoverable — users can find it without instruction
- [ ] Error states and recovery paths are clearly designed

## Consistency

- [ ] Terminology is consistent with the rest of the product (same words for same concepts)
- [ ] Visual patterns are consistent with existing screens
- [ ] Interaction patterns follow established conventions
- [ ] Icon usage is consistent with the icon library

## Usability

- [ ] Primary actions are prominent and easy to find
- [ ] Secondary and destructive actions are appropriately de-emphasised
- [ ] Form validation feedback is clear and actionable
- [ ] Confirmation dialogs are used for destructive actions
- [ ] Users can undo or recover from mistakes where possible

## Mobile

- [ ] Touch targets are ≥44px
- [ ] Text is legible at 375px without horizontal scroll
- [ ] Navigation adapts appropriately for mobile
- [ ] Forms are usable on mobile keyboard (input types are correct)

## Screen States

- [ ] Loading state is designed
- [ ] Empty state is designed with a helpful message and action
- [ ] Error state provides a useful message and recovery path
- [ ] Success state confirms the action and provides next steps
- [ ] Disabled state is visible and labelled

## Accessibility

- [ ] Colour contrast meets WCAG AA minimum (4.5:1 text, 3:1 UI)
- [ ] Design does not rely solely on colour to convey information
- [ ] Focus state is visible for interactive elements
- [ ] Form labels are always visible (not placeholder-only)
- [ ] ARIA roles / labels are noted for screen readers

## Content

- [ ] All copy is final or representative (no Lorem Ipsum)
- [ ] Content length variations are accounted for
- [ ] Error messages are helpful and not technical
- [ ] Call-to-action labels are specific (not generic "Submit")

## Engineering Feasibility

- [ ] Engineering Lead has confirmed approach is technically feasible
- [ ] No design dependencies on features that don't exist yet
- [ ] Performance implications of the design are understood (heavy images, complex animations)

## Background

Nuemart customers currently have no persistent identity artefact they can show in-store. When a customer contacts support, arrives at a delivery, or shops in person, staff must ask for a name or phone number to identify them — a slow and error-prone process.

A customer QR code solves this by giving each customer a scannable identity token tied to their Nuemart account. The QR links to a lightweight Customer ID page that store staff can view instantly on any device.

This feature also lays the groundwork for loyalty tiers, wallet, delivery verification, and coupon QR codes — all of which can reuse the same public token architecture introduced here.

---

## Problem Statement

Store staff and admins have no fast way to identify a Nuemart customer in person. There is no customer ID, membership number, or scannable artefact. This creates friction at the point of sale, at delivery, and in support interactions.

---

## Business Objective

Give every customer a permanent, unique QR code they can show in-store. Enable store staff to scan it and instantly identify the customer. Reduce reliance on name/phone lookup for in-person operations.

## Users / Personas

| Persona | Role | Goal |
|---|---|---|

## Current Flow

There is no current flow. Customers have no profile page and no QR code. Staff must ask for the customer's name or phone number and search manually.

---

## Proposed Flow

### Customer — Viewing and downloading QR code

1. Customer signs in and navigates to account menu.
2. Customer taps "My Profile".
3. Customer Profile page loads (`/account/profile`) showing their name, email, Customer ID, and QR code.
4. If this is the customer's first visit and they do not yet have a `qrCodeId`, the `generateQrCode` mutation fires silently in the background. QR code appears once generated.
5. Customer taps "Download QR" to save the code as an image.

### Staff / Admin — Scanning a customer QR code

1. Staff opens camera or QR scanner app and scans the customer's QR code (shown on phone or printed).
2. QR code encodes the URL: `https://neumart.com/qr/customer/{qrCodeId}`.
3. Browser opens the QR Scan Result page (`/qr/customer/[qrCodeId]`).
4. **Public / unauthenticated view:** Customer name + Customer ID (customerCode) displayed. No other data.
5. **Admin-authenticated view:** Same as above, plus: email / phone (if available), order count, last order date, and a link to the admin customer detail page.

### Failure paths

| Scenario | Expected Behaviour |
|---|---|
| Customer visits `/account/profile` before QR is generated | Page shows a loading / generating state; QR appears within 1–2 seconds after mutation completes |
| `qrCodeId` in URL does not exist in database | Scan result page shows: "QR code not found." No customer data shown. |
| `qrEnabled` is `false` | Scan result page shows: "This QR code has been disabled." No customer data shown. |
| Customer is not signed in and visits `/account/profile` | Redirected to sign-in page (existing auth guard on customer route group) |
| Admin visits scan result page while signed out | Sees public view only (name + customerCode). Enhanced view requires admin sign-in. |

## In Scope

- 4 new optional fields on the `users` Convex table: `customerCode`, `qrCodeId`, `qrEnabled`, `qrCreatedAt`
- Index on `qrCodeId` for fast public lookup
- Auto-generation of `customerCode` and `qrCodeId` for new users inside the existing `syncUser` mutation
- Idempotent `generateQrCode` mutation for existing users (triggered client-side on first profile visit)
- Customer Profile page (`/account/profile`) — auth-gated, shows name, email, customerCode, QR code, download button
- QR Scan Result page (`/qr/customer/[qrCodeId]`) — public, shows limited customer details
- Admin-enhanced view on QR Scan Result page (conditional on authenticated admin role)
- `QrCodeDisplay` component using `react-qr-code`
- Download QR as image (SVG export)
- Account menu link to "My Profile"
- Error states: QR not found, QR disabled, generating state

## Out of Scope

- Rotating or time-limited QR codes
- WhatsApp sharing of QR code
- Apple Wallet / Google Wallet integration
- Printed member card
- Loyalty points or tier system
- Order QR, delivery QR, or coupon QR
- QR scan history or analytics
- Admin UI to disable a customer's QR (field exists in schema; admin UI is a future enhancement)
- Customer ability to regenerate their QR code

## Functional Requirements

| # | Requirement | Priority | Notes |
|---|---|---|---|
| FR-1 | Every new user created via the `syncUser` mutation must have `customerCode` and `qrCodeId` auto-generated at creation time | Must Have | Sequential `CUST-NNNNNN` for customerCode; nanoid (min 10 chars) for qrCodeId |
| FR-2 | Existing users without `qrCodeId` must receive one when they first visit `/account/profile`, via an idempotent `generateQrCode` mutation | Must Have | Mutation must be idempotent — calling it twice must not overwrite existing values |
| FR-3 | Customer Profile page must display: customer name, email, customerCode, and the QR code image | Must Have | QR code rendered via `react-qr-code` |
| FR-4 | The QR code must encode the URL: `https://neumart.com/qr/customer/{qrCodeId}` | Must Have | Must use `qrCodeId`, not Convex `_id` or `tokenIdentifier` |
| FR-5 | The QR Scan Result page must be publicly accessible without authentication | Must Have | No sign-in required to scan a QR code |
| FR-6 | The public `getCustomerByQrCode` Convex query must return only: `name`, `customerCode`, `qrEnabled` | Must Have | Critical security requirement. Must use explicit field selection. Must not return email, phone, role, tokenIdentifier, or Convex _id |
| FR-7 | If the visitor on the QR Scan Result page is a signed-in admin, show additional fields: email (if set), phone (if set), order count, last order date, link to admin customer detail | Must Have | Enhanced view must check admin role server-side or via authenticated Convex query |
| FR-8 | A "Download QR" button must allow the customer to save their QR code as an image | Must Have | SVG export. PNG is acceptable if SVG download is not straightforward. |
| FR-9 | If `qrEnabled` is `false`, the QR Scan Result page must show "This QR code has been disabled" and reveal no customer data | Must Have | |
| FR-10 | If `qrCodeId` is not found in the database, show "QR code not found" with no customer data | Must Have | |
| FR-11 | Account / header navigation must include a **"My QR Code"** link pointing to `/account/profile` | Should Have | Placement: within the existing customer account dropdown or nav menu |
| FR-12 | Customer Profile page must handle the generating state gracefully if QR is being created for the first time | Should Have | Show a spinner or "Generating your QR code..." message while mutation is in-flight |

## Business Rules

| # | Rule |
|---|---|
| BR-1 | `qrCodeId` must be unique across all users. The database and generation logic must enforce this. |
| BR-2 | `customerCode` must be unique across all users. Format: `CUST-` followed by a zero-padded sequential number (e.g. `CUST-000001`). |
| BR-3 | `qrCodeId` must never be the same as the Convex user `_id` or Clerk `tokenIdentifier`. |
| BR-4 | Once assigned, `qrCodeId` and `customerCode` must never change for a given user. |
| BR-5 | The `getCustomerByQrCode` query must never expose `email`, `phone`, `role`, `tokenIdentifier`, or any Convex internal ID to an unauthenticated caller. |
| BR-6 | A disabled QR (`qrEnabled = false`) must return no customer data on the scan result page — not even the customer name. |
| BR-7 | The `generateQrCode` mutation is idempotent: if a user already has `qrCodeId` and `customerCode`, the mutation must return without making any changes. |

## Data Requirements

**New tables:** None.

**New fields on existing `users` table:**

| Field | Type | Required | Default | Notes |
|---|---|---|---|---|
| `customerCode` | `v.optional(v.string())` | No | None | Human-readable customer ID. e.g. `CUST-000124`. Generated once, never changed. |
| `qrCodeId` | `v.optional(v.string())` | No | None | Unique public-safe token for QR URL. e.g. `nq_8F3K92LPA`. Generated once per user using nanoid (min 10 chars). |
| `qrEnabled` | `v.optional(v.boolean())` | No | `true` when generated | Allows QR to be disabled without deletion. Scan result page returns "disabled" state if `false`. |
| `qrCreatedAt` | `v.optional(v.number())` | No | None | Unix timestamp of QR generation. Audit trail only. |

**New index on `users` table:**

```ts
.index("by_qr_code_id", ["qrCodeId"])
```

Required for efficient lookup on the public QR scan result page. Without this index, every QR scan is a full table scan.

## Validations

| Field / Action | Validation Rule | Error Message |
|---|---|---|
| `qrCodeId` (generation) | Must be unique — check before insert | Retry with new nanoid if collision (extremely unlikely but must be handled) |
| `customerCode` (generation) | Must be unique — derived from sequential counter or checked before insert | Retry with next sequence if collision |
| `qrCodeId` (scan result page) | Must exist in database | "QR code not found." |
| `qrEnabled` (scan result page) | Must be `true` | "This QR code has been disabled." |
| `/account/profile` route | User must be authenticated | Redirect to sign-in (handled by existing customer route group auth guard) |

## Edge Cases

| Edge Case | Expected Behaviour |
|---|---|
| User visits `/account/profile` before `generateQrCode` has run (new user, UserSync not yet fired) | Mutation fires client-side on page load; loading state shown until QR is ready |
| Two simultaneous calls to `generateQrCode` for the same user | Mutation is idempotent — second call finds existing values and returns without overwriting |
| `qrCodeId` nanoid collision (extremely rare) | Mutation must check uniqueness before inserting; retry with new nanoid on collision |
| User has `qrEnabled: false` | Scan result page shows disabled message; no name, no customerCode, no data |
| Admin signs out mid-session and refreshes scan result page | Page falls back to public view (name + customerCode only) |
| User with no `name` set in Convex | Profile page shows email only; scan result page shows "Nuemart Customer" or email as fallback |
| User with no `email` set in Convex (edge case — email is a required field in current schema) | Profile page omits email row; scan result admin view omits email row |
| QR code scanned on a device with no internet | Standard browser "no connection" — no special handling required |

## Acceptance Criteria

- [ ] Given a new user signs up, when UserSync runs, then `customerCode` (e.g. `CUST-000001`) and `qrCodeId` (e.g. `nq_8F3K92LPA`) are stored on their Convex user record.
- [ ] Given an existing user without a QR code visits `/account/profile`, when the page loads, then `generateQrCode` fires and their QR code appears within 2 seconds.
- [ ] Given a user visits `/account/profile`, when the page renders, then their name, email, customerCode, and a scannable QR code are all visible.
- [ ] Given the QR code is rendered, when it is scanned with a phone, then the browser opens `https://neumart.com/qr/customer/{qrCodeId}`.
- [ ] Given an unauthenticated visitor opens the QR scan result page, then only the customer's name and customerCode are shown — no email, phone, or sensitive data.
- [ ] Given an admin opens the QR scan result page while signed in, then email, phone (if available), order count, last order date, and an admin link are also shown.
- [ ] Given the `getCustomerByQrCode` query is called, then it never returns `email`, `phone`, `role`, `tokenIdentifier`, or Convex `_id` fields.
- [ ] Given a `qrCodeId` that does not exist is accessed, then the scan result page shows "QR code not found" with no customer data.
- [ ] Given a user with `qrEnabled: false`, when their QR is scanned, then the page shows "This QR code has been disabled" with no customer data.
- [ ] Given the customer taps "Download QR", then a QR code image file is saved to their device.
- [ ] Given `generateQrCode` is called twice for the same user, then the second call makes no changes and the original `qrCodeId` and `customerCode` are preserved.

## Rollout Plan

1. Install `react-qr-code` package.
2. Run `npx convex deploy` — schema migration (4 optional fields + index on users table) is automatic.
3. Deploy Next.js app — 2 new pages and 1 new component; no existing pages modified except account nav.
4. No feature flag required — new screens are additive and independent.
5. No data backfill required — existing users receive QR codes lazily on first profile visit.
6. Post-deploy: smoke test QR generation, scan, and download on a test customer account.

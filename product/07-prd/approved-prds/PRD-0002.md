## Background

The Nuemart storefront currently opens directly to a product grid with a category filter bar. There is no visual merchandising area at the top of the page — no way to highlight offers, seasonal collections, or featured categories beyond the product grid itself.

Grocery apps in the Indian market (Blinkit, Zepto, BigBasket, Amazon Fresh) universally use a promotional banner carousel at the top of their storefront to communicate offers, drive category awareness, and improve first-impression conversion. This is a standard and expected pattern for Indian grocery e-commerce.

---

## Problem Statement

The Nuemart storefront has no visual merchandising surface. Customers land on a plain product grid with no awareness of offers, seasonal collections, or featured categories. A promotional banner carousel at the top of the product listing page addresses this gap and aligns the storefront with industry norms.

---

## Business Objective

- Improve the first-impression experience of the Nuemart storefront.
- Create a visual merchandising surface to highlight offers, seasonal items, and featured categories.
- Bring the storefront in line with standard Indian grocery e-commerce UX expectations.
- Lay groundwork for a future admin-managed banner system with zero UI rewrite.

## Current Flow

1. Customer navigates to `/products`.
2. Sees the category filter bar and product grid immediately — no visual merchandising.
3. No offers or seasonal promotions are surfaced above the fold.

---

## Proposed Flow

**Happy path — Customer views carousel:**

1. Customer navigates to `/products`.
2. Carousel renders at the top of the page, above the category filter and product grid.
3. First active banner is shown. Auto-scrolls to the next banner every 4 seconds.
4. Dot indicators show the customer which banner they are on and how many total banners exist.
5. On desktop: left and right arrow buttons allow manual navigation.
6. On mobile: dots are visible; customer can swipe left/right to navigate manually; arrows are hidden.
7. Customer clicks a banner → navigates to the `linkTarget` route (e.g. a category page or product listing with a filter).
8. Auto-scroll pauses while the customer is actively interacting (hovering on desktop, touching on mobile) and resumes after interaction ends.

**Failure paths:**

| Scenario | Expected Behaviour |
|---|---|
| All banners have `isActive: false` | Carousel does not render; product grid shows at top as normal |
| Only one active banner | Carousel renders with that single banner; no dots shown; no arrows shown; no auto-scroll |
| Image URL returns 404 / broken | Banner renders with a fallback background colour; no broken image icon visible |
| `linkTarget` is an empty string or undefined | Banner is non-clickable; no navigation occurs; no error thrown |

## In Scope

- `BannerConfig` TypeScript type definition with all required fields.
- Static banner config array in a dedicated config file (`lib/banners.config.ts` or `config/banners.ts`).
- `<BannerCarousel>` React component that accepts `banners: BannerConfig[]` as a prop.
- Auto-scroll with configurable interval (default 4 seconds).
- Manual navigation via arrow buttons (desktop) and swipe gesture (mobile).
- Dot indicators on all screen sizes.
- Arrow buttons visible on desktop, hidden on mobile via CSS breakpoint.
- Responsive design: mobile-first, rounded corners, good spacing.
- Fallback behaviour when no active banners exist (carousel hidden, no layout shift).
- Click-through navigation via `linkTarget` route using Next.js `router.push`.
- Integration into `app/(customer)/products/page.tsx` above the category filter and product grid.
- Placeholder banner images with representative promotional text for MVP.

---

## Out of Scope

- Admin-managed banners (create, edit, reorder, activate/deactivate, start/end dates) — Future module candidate.
- A new Convex `banners` table — Not required for MVP; static config only.
- Admin screen for banner management — Future candidate.
- Animated text overlays or video banners.
- A/B testing or personalised banner targeting.
- Analytics or click tracking per banner.
- Deep-linking to specific products (only category or listing routes allowed as `linkTarget` for MVP).
- Push notifications tied to banner content.

## Business Rules

| # | Rule |
|---|---|
| BR-1 | Banners are displayed in the order they appear in the config array (index 0 first) |
| BR-2 | Only `isActive: true` banners are passed to the carousel component |
| BR-3 | The config file is the single source of truth for MVP banner content |
| BR-4 | Banner images must have a defined aspect ratio (16:5 recommended) to prevent layout shift during load |
| BR-5 | `linkTarget` must be an internal route (starts with `/`). External links are not supported in MVP |

## Data Requirements

**New Convex tables:** None for MVP.

**Fields on existing tables:** None.

**New frontend config type:**

```typescript
// lib/banners.config.ts (or config/banners.ts)

export type BannerConfig = {
  id: string;           // Unique identifier — used as React key and future DB primary key
  imageUrl: string;     // Banner image URL
  title?: string;       // Optional headline text overlay
  subtitle?: string;    // Optional subtext overlay
  ctaText?: string;     // Optional CTA button label (e.g. "Shop Now")
  linkTarget: string;   // Internal route to navigate to on click (e.g. "/products?category=fruits")
  isActive: boolean;    // Whether to show this banner
};

export const banners: BannerConfig[] = [
  // MVP: 3–5 placeholder banners with representative content
];
```

> **Scalability note:** The `BannerConfig` type is intentionally shaped to mirror a future Convex `banners` table. When the admin-managed banner module is built, the carousel component requires zero changes — only the data source (prop value) changes from the static config to a Convex `useQuery` result.

## Edge Cases

| Edge Case | Expected Behaviour |
|---|---|
| Zero active banners | Carousel component not rendered; no empty space or placeholder |
| One active banner | Carousel renders with that banner only; no arrows, no dots, no auto-scroll |
| Broken image URL | Banner renders with a CSS fallback background colour (e.g. `bg-muted`); no broken image icon |
| Banner with no `title` or `ctaText` | Banner renders as image-only; no overlay elements render |
| User navigates away and back while auto-scroll is mid-cycle | Timer resets on mount; no stale interval |
| `linkTarget` is `/` (home) | Navigates to home/root; no error |
| Very long `title` or `subtitle` | Text truncates or wraps within the overlay; does not overflow the banner bounds |
| Slow image load | Image fades in after load; layout does not shift (aspect ratio container holds space) |

## Acceptance Criteria

- [ ] Given the customer is on `/products`, when the page loads, the banner carousel renders above the category filter and product grid with at least one active banner visible.
- [ ] Given the carousel has more than one active banner, when 4 seconds pass, the carousel auto-scrolls to the next banner.
- [ ] Given the carousel is visible on desktop, when the customer clicks the right arrow, the next banner is shown.
- [ ] Given the carousel is visible on desktop, when the customer clicks the left arrow, the previous banner is shown.
- [ ] Given the carousel is on mobile, when the customer swipes left, the next banner is shown.
- [ ] Given the carousel is on mobile, the arrow buttons are not visible.
- [ ] Given the carousel has multiple banners, the dot indicators correctly reflect the currently active banner.
- [ ] Given the customer clicks a banner with a valid `linkTarget`, the customer is navigated to that route without a full page reload.
- [ ] Given a banner has `isActive: false`, that banner does not appear in the carousel.
- [ ] Given all banners have `isActive: false`, the carousel section does not render and the category filter bar appears at the top.
- [ ] Given a banner image URL is broken, the banner renders with a fallback background colour and no broken image icon is shown.
- [ ] Given only one active banner exists, no arrows and no dots are rendered, and auto-scroll is disabled.

## Rollout Plan

Single deployment. No feature flag required. Change is purely additive — the carousel is inserted above the existing product grid and does not modify any existing behaviour. If the carousel needs to be disabled quickly, all banners can be set to `isActive: false` in the config with a redeployment.

## Background

Nuemart currently shows only basic product information on both the product listing and product detail pages: name, quantity, price, image, stock status, and add-to-cart. There is no structured way for the product catalogue to carry allergen or ingredient information.

Customers shopping for groceries often have dietary restrictions, food allergies, religious food requirements, or personal preferences (vegetarian, vegan, Jain, low fat, etc.) that require them to know what is inside a product before they buy. Without this information in the app, customers must either look at physical packaging (not possible online), call the store, or guess — all of which reduce trust and increase abandonment.

This PRD adds a structured allergen and ingredient layer to the product catalogue. Admin can capture this data per product. Customers see a compact indicator on the product card and full details on the product detail page.

---

## Problem Statement

Customers cannot make informed purchase decisions about allergens, ingredients, or dietary suitability from the current product pages. Nuemart has no mechanism to capture or display this information. This is a product transparency gap that creates friction for allergy-conscious, diet-conscious, and health-conscious shoppers.

---

## Business Objective

- Reduce purchase uncertainty for customers with dietary restrictions or food allergies.
- Increase customer trust by showing transparency about product contents.
- Lay the schema foundation for future allergen/dietary filtering (REQ-0006, Parked).
- Create a structured, FSSAI-aligned product data layer as a long-term compliance asset.

## Current Flow

### Customer — Product Detail

1. Customer opens product listing page (`/products`)
2. Customer clicks on a product card
3. Customer sees: product image, name, quantity, price, description, stock status, add-to-cart button
4. **No ingredient or allergen information is shown anywhere**
5. Customer cannot determine suitability without external research

### Admin — Product Create / Edit

1. Admin navigates to `/admin/products/new` or `/admin/products/[id]/edit`
2. Admin fills in: name, slug, description, price, category, image, stock, active status
3. **No allergen or ingredient fields exist on the form**
4. Admin has no way to record this information in the system

---

## Proposed Flow

### Customer — Product Listing (SCR-CUS-0001)

1. Customer opens `/products`
2. Each product card renders as before
3. **If the product has `containsAllergens` set:** a badge reads "Contains allergens"
4. **If the product has only `mayContainAllergens` set:** a badge reads "May contain allergens"
5. **If both are set:** badge reads "Contains allergens" only
6. **If no allergen data exists:** no badge is shown
7. Badge is display-only — no click, no tooltip, no modal

### Customer — Product Detail (SCR-CUS-0002)

1. Customer clicks a product card and opens `/products/[slug]`
2. Existing product content renders: image, name, price, description, add-to-cart
3. Below the description, the **Allergen & Ingredient Information** section renders
4. **If allergen data exists:** section shows all populated sub-sections (Ingredients, Contains, May Contain, Dietary Tags, Notes)
5. **If no allergen data exists:** section renders with only: "Allergen information not available."
6. Sub-sections with no data are not rendered individually — only populated sub-sections appear

### Admin — New / Edit Product (SCR-ADM-0006, SCR-ADM-0007)

1. Admin opens `/admin/products/new` or `/admin/products/[id]/edit`
2. Existing product fields render as before
3. A new **Allergen & Ingredient** section appears at the bottom of the form
4. Admin can fill in any combination of the 5 fields or leave all blank
5. Admin saves — all filled allergen fields are persisted; blank fields are saved as undefined
6. On edit: existing allergen data is pre-populated in the form

**Failure paths:**

| Scenario | Expected Behaviour |
|---|---|
| Admin saves product with all allergen fields blank | Product saves successfully. Allergen fields stored as undefined. |
| Admin clears previously set allergen data | Fields saved as undefined. Customer detail page shows "Allergen information not available." |
| Customer views product with no allergen data | "Allergen information not available." is shown. No allergen badge on card. |
| Admin enters an allergen not in the fixed list | Not possible — input is a multi-select from a fixed list. Free text not accepted. |
| Admin enters a dietary tag not in the fixed list | Not possible — same constraint. |

## In Scope

1. 5 new optional fields on the `products` schema: `ingredients`, `containsAllergens`, `mayContainAllergens`, `dietaryTags`, `allergenNotes`
2. Admin product create form (SCR-ADM-0006): 5 new optional fields with appropriate input types
3. Admin product edit form (SCR-ADM-0007): same 5 fields, pre-populated from existing data
4. Product card (SCR-CUS-0001): generic allergen badge when allergen data is present
5. Product detail page (SCR-CUS-0002): full Allergen & Ingredient Information section
6. Fallback message: "Allergen information not available." when no data is present
7. Fixed multi-select lists for allergens (12 items) and dietary tags (10 items)
8. Schema designed as arrays for future filter-readiness

---

## Out of Scope

1. **Allergen / dietary tag filtering on the product listing page** — tracked as REQ-0006 (Parked; reopen when REQ-0005 ships and catalogue data is populated)
2. **Admin completeness indicator** (Missing / Added / Needs review per product) — Future Candidate in MODULE_MASTER.md
3. **Custom / freeform dietary tags** — Future Candidate; requires admin-managed tag master
4. **Allergen data auto-import** from supplier feeds or product barcodes
5. **Customer allergen profile** (save my allergies, warn me on every product) — not scoped
6. **Allergen search or exclusion filters** — part of REQ-0006
7. **Regulatory compliance reporting** — not in scope; data structure supports it as a future step

## Business Rules

| # | Rule |
|---|---|
| BR-1 | If `containsAllergens` has one or more values, the product card badge reads **"Contains allergens"** |
| BR-2 | If `containsAllergens` is absent or empty AND `mayContainAllergens` has one or more values, the product card badge reads **"May contain allergens"** |
| BR-3 | If both `containsAllergens` and `mayContainAllergens` are set, the badge reads **"Contains allergens"** only — `containsAllergens` takes precedence |
| BR-4 | If both `containsAllergens` and `mayContainAllergens` are absent or empty, no badge is shown on the product card |
| BR-5 | On the product detail page, if no allergen-related fields are populated (all five are undefined/absent), the section shows only: "Allergen information not available." |
| BR-6 | On the product detail page, a sub-section (e.g. "Ingredients") is only rendered when its field has data. An empty `ingredients` field does not render the Ingredients sub-section. |
| BR-7 | The text "No allergens", "Allergen-free", "Safe for allergies" or any equivalent affirmative claim must never be auto-generated by the system |
| BR-8 | `containsAllergens` and `mayContainAllergens` values must only be drawn from the fixed allergen list — never freetext |
| BR-9 | `dietaryTags` values must only be drawn from the fixed dietary tag list — never freetext |
| BR-10 | Admin can set any combination of allergen fields independently — there is no enforced dependency between fields |

## Screen Impact

### SCR-CUS-0001 — Product Listing (`/products`)

**Change:** Add allergen badge to product card component (`product-card.tsx`).

Badge rendering logic:
- `containsAllergens?.length > 0` → render badge: "Contains allergens"
- `mayContainAllergens?.length > 0` (and no containsAllergens) → render badge: "May contain allergens"
- Neither condition met → render no badge

Badge design: small, non-interactive label — consistent with existing badge styles in the product card (e.g. "Out of Stock" badge treatment). No icon required in MVP.

---

### SCR-CUS-0002 — Product Detail (`/products/[slug]`)

**Change:** Add `AllergenInfoSection` component below the product description.

Section structure (rendered only when data is present for each sub-section):

```
Allergen & Ingredient Information

Ingredients
[value of ingredients field]

Contains
[comma-separated list of containsAllergens values]

May Contain Traces Of
[comma-separated list of mayContainAllergens values]

Dietary Information
[comma-separated list of dietaryTags values]

Notes
[value of allergenNotes field]
```

Fallback (when all five fields are absent/undefined):

```
Allergen & Ingredient Information

Allergen information not available.
```

The section heading ("Allergen & Ingredient Information") always renders. Only sub-sections with data render below it.

---

### SCR-ADM-0006 — New Product (`/admin/products/new`)

**Change:** Add a collapsible or visually separated "Allergen & Ingredient" section at the bottom of the product form.

Fields:
1. **Ingredients** — textarea, optional, placeholder: "e.g. Wheat flour, sugar, salt, edible vegetable oil"
2. **Contains Allergens** — multi-select, optional, values from fixed allergen list
3. **May Contain Allergens** — multi-select, optional, values from fixed allergen list
4. **Dietary Tags** — multi-select, optional, values from fixed dietary tag list
5. **Allergen Notes** — textarea, optional, placeholder: "e.g. Manufactured in a facility that also processes nuts"

---

### SCR-ADM-0007 — Edit Product (`/admin/products/[id]/edit`)

**Change:** Same 5 fields as New Product form. Existing product allergen data pre-populated from Convex. Saving the form patches only the allergen fields that are submitted; other product fields are unaffected.

## Edge Cases

| Edge Case | Expected Behaviour |
|---|---|
| Product has `containsAllergens: []` (empty array, not undefined) | Treat same as absent — no badge shown, no "Contains" sub-section rendered |
| Product has `mayContainAllergens: []` (empty array, not undefined) | Treat same as absent — no badge shown, no "May Contain" sub-section rendered |
| Admin sets `containsAllergens` then later clears it (deselects all) | Field saved as undefined or empty array; badge removed from card; section no longer renders Contains sub-section |
| Product has `dietaryTags` but no allergen fields | No allergen badge on card (dietary tags do not trigger badge); detail page shows only the Dietary Information sub-section |
| Admin adds `allergenNotes` but no other allergen fields | Detail page shows only the Notes sub-section; no badge on card; other sub-sections not rendered |
| Very long `ingredients` string | Rendered in a scrollable or wrapped text block; no truncation or character limit enforced in MVP |

## Acceptance Criteria

- [ ] Given a product with `containsAllergens: ["Milk", "Wheat"]`, when a customer views the product listing, then the product card shows a "Contains allergens" badge
- [ ] Given a product with no `containsAllergens` and `mayContainAllergens: ["Peanuts"]`, when a customer views the product listing, then the product card shows a "May contain allergens" badge
- [ ] Given a product with both `containsAllergens` and `mayContainAllergens` set, when a customer views the product listing, then the card shows "Contains allergens" only (not both badges)
- [ ] Given a product with no allergen data, when a customer views the product listing, then no allergen badge appears on the card
- [ ] Given a product with full allergen data, when a customer views the product detail page, then all five sub-sections (Ingredients, Contains, May Contain, Dietary Information, Notes) are displayed
- [ ] Given a product with only `ingredients` and `dietaryTags` set, when a customer views the product detail page, then only the Ingredients and Dietary Information sub-sections are rendered — Contains and May Contain sub-sections are not shown
- [ ] Given a product with no allergen data, when a customer views the product detail page, then the section header renders with only "Allergen information not available." — no empty sub-section headers appear
- [ ] Given any product without allergen data, the system never displays "No allergens", "Allergen-free", or any affirmative safety claim
- [ ] Given an admin creates a product with all allergen fields blank, when the form is submitted, then the product saves without error
- [ ] Given an admin fills in the Contains Allergens field, when they submit the form, then only values from the fixed allergen list are accepted — no freetext is possible
- [ ] Given an admin fills in Dietary Tags, when they submit, then only values from the fixed dietary tag list are accepted
- [ ] Given an existing product, when an admin opens the edit form, then existing allergen data is pre-populated in the form fields
- [ ] Given an admin edits a product's allergen data, when the form is saved, then the updated allergen data is reflected immediately on the product card and product detail page

5. Post-release: admin team to progressively enrich product catalogue with allergen data. This is a manual data entry effort — no system action required.

No feature flags required. No staged rollout required. Purely additive change with no risk to existing functionality.

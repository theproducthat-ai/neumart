## Background

Nuemart is a grocery ordering platform where every order placed by a customer requires physical delivery. As of Phase 10, the platform has a complete browse-to-order flow — customers can browse, cart, checkout, and view their order history. However, no delivery management capability exists. Once an order is placed, the admin tracks delivery manually via phone calls and WhatsApp. The customer has no system visibility into whether their order is being prepared, dispatched, or delivered.

This PRD specifies the first version of Delivery Management for Nuemart: a lightweight, admin-managed delivery tracking layer that is fully decoupled from order status, payment, and inventory.

---

## Problem Statement

After a customer places an order on Nuemart, neither the customer nor the admin has any system visibility into the delivery lifecycle. The admin tracks delivery manually with no audit trail, and the customer has no way to know the current status of their delivery short of calling the store.

---

## Business Objective

1. Give the admin a structured delivery workflow embedded in the existing Admin Order Detail screen — eliminating ad-hoc WhatsApp/phone tracking and creating an audit trail per order.
2. Give the customer read-only delivery status visibility on their Order Detail page — reducing "where is my order?" support contacts.
3. Capture delivery data from the first real order so that future delivery capabilities (rider app, proof of delivery, delivery reports) can be built on a solid data foundation.

---

## Users / Personas

| Persona | Role | Goal |
|---|---|---|
| Store customer | Customer | Know the current delivery status of their placed order without calling the store |
| Store owner / operations staff | Admin | Assign a delivery person, track delivery progress, and log outcomes (delivered or failed) with a record per order |
| Delivery person | Not a system actor in MVP | Handled by admin on their behalf. Future phase only. |

---

## Current Flow

**Admin (today):**
1. Customer places order.
2. Admin sees order in Admin Order List.
3. Admin calls or WhatsApps a delivery person to arrange pickup.
4. Delivery person collects and delivers — no system record.
5. Admin manually updates the order status to "Delivered" with no delivery-specific data.

**Customer (today):**
1. Customer places order.
2. Customer can view order history and order detail — sees order status only.
3. Customer has no visibility into delivery progress. May call the store to ask.

---

## Proposed Flow

### Admin — Happy Path

1. Customer places order → system automatically creates a `deliveryTasks` record linked to that order, with status = **Pending**.
2. Admin opens Admin Order Detail (SCR-ADM-0009) → sees a **Delivery** section showing status "Pending".
3. Admin enters delivery person name and contact number → clicks **Assign** → status updates to **Assigned**. Assignment timestamp recorded.
4. Admin updates status to **Picked Up** when the delivery person collects the order. Timestamp recorded.
5. Admin updates status to **Out for Delivery** when en route. Timestamp recorded.
6. Admin updates status to **Delivered** on confirmation. Timestamp recorded.
7. At any point, admin can update status to **Cancelled** with an optional reason.

### Admin — Failed Delivery Path

1. Delivery attempt occurs but fails.
2. Admin updates status to **Failed**.
3. Admin selects a reason from a fixed dropdown (see FR-10). If "Other" is selected, a notes field is shown and is mandatory.
4. Failed delivery reason and timestamp are saved.
5. Order status, payment status, and stock are **unchanged**.

### Customer — Happy Path

1. Customer opens Order Detail (SCR-CUS-0010) → sees a **Delivery Status** section.
2. Section shows: current delivery status label, delivery address.
3. Status updates in real time as admin updates the delivery task.
4. Customer does NOT see delivery person name, contact, or internal notes.

### Failure Paths

| Scenario | Expected Behaviour |
|---|---|
| Admin sets status = Failed without selecting a reason | Form validation blocks submission. Error: "Please select a failed delivery reason." |
| Admin selects "Other" reason with no notes | Form validation blocks submission. Error: "Please add notes when selecting Other." |
| Admin attempts invalid status transition | Server-side mutation rejects it. Error: "Cannot set delivery to [X] from current status [Y]." |
| Order placed but delivery task creation fails | Error is surfaced — not silently swallowed. Order mutation must not complete if delivery task creation fails. |
| Customer views order where delivery is Failed | Status shows "Delivery Failed". No reason or notes shown to customer. |
| Customer views order where delivery is Cancelled | Status shows "Delivery Cancelled". No reason shown to customer. |

---

## In Scope

1. `deliveryTasks` Convex table with full schema (see Data Requirements)
2. Automatic delivery task creation (status = Pending) as part of order placement mutation
3. Admin: view delivery task status in Admin Order Detail
4. Admin: assign delivery person (name + contact) — transitions status to Assigned
5. Admin: update delivery status through 7-state lifecycle
6. Admin: capture failed delivery reason via fixed dropdown + optional notes (mandatory for "Other")
7. Admin: cancel delivery with optional reason
8. Customer: read-only delivery status + delivery address on Customer Order Detail
9. Delivery status is fully independent of order status, payment status, and stock

---

## Out of Scope

The following are explicitly deferred to future iterations and must not be built as part of this PRD:

- Standalone `/admin/delivery` list screen
- Delivery person login or mobile self-service interface
- Automated customer notifications (SMS, WhatsApp, email, push)
- Proof of delivery (OTP, photo, signature)
- Third-party delivery partner API integration (Dunzo, Shadowfax, Porter, etc.)
- Live map or GPS tracking
- Delivery fee calculation
- Delivery SLA monitoring or reporting
- Failed delivery retry, reassignment, or cancellation workflows
- Pickup / self-collection fulfilment type
- `deliveryPersons` Convex table (delivery person is stored as inline name/contact strings in MVP)

## Functional Requirements

| # | Requirement | Priority | Notes |
|---|---|---|---|
| FR-1 | When a customer successfully places an order, the system automatically creates a `deliveryTasks` record with `status = "pending"`, linked to the `orderId`, `userId`, and `addressId`. | Must Have | Atomic with order placement. Not a separate async call. |
| FR-2 | Admin can view the delivery status of any order in the Delivery section of Admin Order Detail (SCR-ADM-0009). | Must Have | Shows status, assigned person if any, and all relevant timestamps. |
| FR-3 | Admin can assign a delivery person (name and optional contact number) to a delivery task. Assigning transitions status from Pending to Assigned and records `assignedAt`. | Must Have | `assignedTo` is required to assign; `assignedContact` is optional. |
| FR-4 | Admin can update delivery status through the 7-state lifecycle: Pending → Assigned → Picked Up → Out for Delivery → Delivered or Failed or Cancelled. | Must Have | Each transition records a timestamp. Validated server-side. |
| FR-5 | Admin can only transition status in the valid sequence defined in BR-6. Invalid transitions are rejected server-side with a clear error message. | Must Have | Enforced in the Convex mutation, not only in the UI. |
| FR-6 | Admin must select a failed delivery reason from a fixed dropdown when setting status to Failed. | Must Have | 8 options (see FR-10). Reason is required — form cannot submit without it. |
| FR-7 | If admin selects "Other" as the failed delivery reason, a notes field is shown and is mandatory. | Must Have | Notes are optional for all other reason codes. |
| FR-8 | Setting delivery status to Failed does not change order status, payment status, or stock. | Must Have | Three hard independence rules. Enforced in server-side mutation logic. |
| FR-9 | Setting delivery status to any value does not change order status, payment status, or stock. | Must Have | The delivery mutation must never write to `orders`, `payments`, or `products` tables. |
| FR-10 | Failed delivery reason dropdown options: Customer unavailable / Wrong address / Customer refused delivery / Delivery person unavailable / Order damaged / Payment issue / Unable to contact customer / Other. | Must Have | Stored as a string enum on `deliveryTasks.failedReasonCode`. |
| FR-11 | Admin can cancel a delivery task (status → Cancelled) with an optional cancellation reason. Cancelling delivery does not change order status, payment, or stock. | Must Have | |
| FR-12 | Customer can see a Delivery Status section on their Order Detail page (SCR-CUS-0010) showing the current delivery status label and delivery address. | Must Have | Read-only. Visible from the moment an order is placed (Pending status). |
| FR-13 | Customer does not see delivery person name, contact number, internal notes, or failed delivery reason text. | Must Have | Failed status shows "Delivery Failed" label only. |
| FR-14 | A customer can only view the delivery task for their own order. The Convex query must verify `deliveryTask.userId === authenticated user's ID`. | Must Have | Server-side check. Never trust client-supplied userId (DEC-003). |
| FR-15 | All write mutations (assign, update status, failed reason, cancel) are restricted to admin role only. Verified server-side via Clerk `publicMetadata.role === "admin"`. | Must Have | |
| FR-16 | Each order has at most one delivery task. Attempting to create a second delivery task for the same order must be rejected. | Must Have | Enforced by a unique-by-orderId constraint or check in the mutation. |
| FR-17 | `updatedAt` and `updatedBy` are recorded on the delivery task on every status change. | Must Have | Provides a basic audit trail. |

---

## Business Rules

| # | Rule |
|---|---|
| BR-1 | Delivery status is independent of order status. Updating delivery status must NEVER change `orders.status`. |
| BR-2 | Delivery status updates must NEVER change `payments.status` or any field on the `payments` table. |
| BR-3 | Delivery status updates must NEVER change `products.stock` or create `stockMovements` records. |
| BR-4 | Admin is the only role that can assign a delivery person or update delivery status in MVP. |
| BR-5 | Each order has exactly one delivery task. One-to-one relationship enforced. |
| BR-6 | Valid status transitions only: Pending → Assigned; Assigned → Picked Up; Picked Up → Out for Delivery; Out for Delivery → Delivered; Out for Delivery → Failed; Out for Delivery → Cancelled; Any status → Cancelled (admin override). Any other transition is rejected. |
| BR-7 | `failedReasonCode` is required when status = Failed. `failedReasonNotes` is required when `failedReasonCode = "other"`. |
| BR-8 | Failed delivery must NOT automatically cancel the order, change payment status, or restock inventory. Admin handles follow-up manually. |
| BR-9 | Delivery person is assigned as inline name (string) and contact (string). No foreign key to a `deliveryPersons` table in MVP. The schema is designed to allow future migration to a FK-based model without breaking changes. |
| BR-10 | Customer sees delivery status and delivery address only. Delivery person name/contact, internal notes, and admin-only failed reason text are never exposed to customers. |
| BR-11 | Delivery task creation must be atomic with order placement. If delivery task creation fails, the order must not be created either. Silent failures are not acceptable. |
| BR-12 | Delivery task is created with `createdAt` set to the order placement timestamp. `addressId` stored on the delivery task links to the address selected at checkout. |

## Data Requirements

### New Convex Table: `deliveryTasks`

| Field | Type | Required | Notes |
|---|---|---|---|
| `orderId` | `Id<"orders">` | Yes | Links to the parent order. Index: `by_orderId`. One per order. |
| `userId` | `Id<"users">` | Yes | The customer who placed the order. Used for authorization checks. |
| `addressId` | `Id<"addresses">` | Yes | The delivery address selected at checkout. |
| `status` | string literal union | Yes | `"pending"` \| `"assigned"` \| `"picked_up"` \| `"out_for_delivery"` \| `"delivered"` \| `"failed"` \| `"cancelled"` |
| `assignedTo` | string | No | Delivery person name. Required when transitioning to "assigned". |
| `assignedContact` | string | No | Delivery person phone number. Optional. |
| `assignedAt` | number | No | Unix timestamp. Set when status → assigned. |
| `pickedUpAt` | number | No | Unix timestamp. Set when status → picked_up. |
| `outForDeliveryAt` | number | No | Unix timestamp. Set when status → out_for_delivery. |
| `deliveredAt` | number | No | Unix timestamp. Set when status → delivered. |
| `failedAt` | number | No | Unix timestamp. Set when status → failed. |
| `failedReasonCode` | string literal union | No | `"customer_unavailable"` \| `"wrong_address"` \| `"customer_refused"` \| `"delivery_person_unavailable"` \| `"order_damaged"` \| `"payment_issue"` \| `"unable_to_contact"` \| `"other"`. Required when status = failed. |
| `failedReasonNotes` | string | No | Required when `failedReasonCode = "other"`. Optional for all other codes. |
| `cancelledAt` | number | No | Unix timestamp. Set when status → cancelled. |
| `cancelledReason` | string | No | Optional admin note on cancellation. |
| `createdAt` | number | Yes | Unix timestamp. Set at order placement. |
| `updatedAt` | number | Yes | Unix timestamp. Updated on every status change. |
| `updatedBy` | `Id<"users">` | Yes | The admin user who last changed the status. Never accepted from client — resolved from `ctx.auth`. |

**Index required:** `by_orderId` on `deliveryTasks` — enables efficient delivery task lookup for a given order.

### Existing Tables Affected

| Table | Change |
|---|---|
| `orders` | No new fields. Order placement mutation modified to also call delivery task creation. Read-only from delivery module's perspective thereafter. |
| `schema.ts` | Add `deliveryTasks` table definition and `by_orderId` index. |

## Edge Cases

| Edge Case | Expected Behaviour |
|---|---|
| Order placed but delivery task creation fails | Entire order placement transaction fails. Error returned to user. No silent failure. |
| Admin attempts to assign without entering a name | Form validation blocks it. "Delivery person name is required to assign." |
| Admin sets Failed, selects "Other", leaves notes blank | Form validation blocks it. "Please add notes when selecting Other." |
| Admin tries to transition to an invalid status | Server-side mutation rejects. Error: "Cannot set delivery to [X] from current status [Y]." |
| Admin tries to create a second delivery task for the same order | Mutation checks for existing record and rejects. |
| Customer visits order where no delivery task exists | Should not occur (task created at order placement). If it does, Delivery section shows a neutral "Not available" state. |
| Customer tries to query another customer's delivery task | `getDeliveryTaskByOrder` resolves user from `ctx.auth`, not from any client argument. Returns null if `deliveryTask.userId` does not match. |
| Delivery status = Failed, admin views on admin order detail | Shows full failed reason + notes. Admin-only. |
| Delivery status = Failed, customer views on order detail | Shows "Delivery Failed" label only. No reason. |
| Admin assigns delivery person and immediately cancels | Allowed. Status → Cancelled. `assignedTo` remains stored on the record. |

## Reporting

**MVP scope:** No new dashboard stats or reporting screens.

**Audit trail:** Every delivery task status change is timestamped with `updatedAt` and `updatedBy`. This provides a per-order delivery history that can be surfaced in future reporting.

**Future:** Delivery metrics (delivered %, failed %, average delivery time, failed reason breakdown) can be derived from the `deliveryTasks` table once sufficient data exists.

## Non-Functional Requirements

| Requirement | Specification |
|---|---|
| Real-time updates | Delivery status on both customer and admin screens must update in real time via Convex's reactive query system (`useQuery`). No manual refresh required. |
| Authorization | All delivery write mutations must resolve the caller identity from `ctx.auth` (never from client arguments — DEC-003). Admin role check server-side. Customer ownership check server-side. |
| Schema type safety | `deliveryTasks.status` and `deliveryTasks.failedReasonCode` must be defined as Convex literal union types in `schema.ts`, not raw strings. This enforces valid values at the schema level. |
| Atomicity | Delivery task creation must be part of the same Convex mutation as order placement (or called synchronously within it). Not fire-and-forget. |
| No cross-contamination | The delivery mutation must never write to `orders`, `payments`, `stockMovements`, or `products` tables. |
| Rollback safety | Adding `deliveryTasks` is additive-only. No existing table is modified structurally. Rollback is a schema revert + file deletion. |

## Acceptance Criteria

- [ ] Given a customer places an order, then a `deliveryTasks` record is created with `status = "pending"`, linked to the correct `orderId`, `userId`, and `addressId`.
- [ ] Given a delivery task exists, when admin opens Admin Order Detail, then the Delivery section is visible with the current status.
- [ ] Given delivery status is Pending, when admin enters a delivery person name and clicks Assign, then status updates to "assigned", `assignedTo` and `assignedAt` are saved.
- [ ] Given a delivery task exists, when admin updates the status, then `updatedAt` and `updatedBy` (the admin's user ID) are recorded.
- [ ] Given a delivery task is Assigned or later, when customer opens Order Detail, then the Delivery Status section shows the current status label and delivery address.
- [ ] Given customer views order detail, then delivery person name and contact are NOT visible.
- [ ] Given admin sets status to Failed with no reason selected, then form submission is blocked with the error "Please select a failed delivery reason."
- [ ] Given admin sets status to Failed with reason = "Other" and no notes, then form submission is blocked with the error "Please add notes when selecting Other."
- [ ] Given admin sets status to Failed with a valid reason, then `failedReasonCode`, `failedReasonNotes`, and `failedAt` are saved.
- [ ] Given delivery status is set to Failed, then `orders.status` is unchanged, `payments.status` is unchanged, and `products.stock` is unchanged.
- [ ] Given admin attempts an invalid status transition, then the server rejects it with the error "Cannot set delivery to [X] from current status [Y]."
- [ ] Given one delivery task already exists for an order, when admin tries to create another, then the mutation rejects it.
- [ ] Given a customer tries to view a delivery task for another customer's order, then the query returns null (no data exposed).
- [ ] Given delivery task creation fails during order placement, then the entire order placement fails — no orphaned order exists without a delivery task.

## What Is Being Evaluated

The Delivery Management module adds a delivery lifecycle layer to Nuemart's existing order flow. Currently, once an order is placed, neither the admin nor the customer has any system-level visibility into delivery — fulfilment is tracked manually via phone and WhatsApp with no audit trail. This evaluation assesses whether to build the first version of this module now, and at what scope, given the current state of the platform (Pay Later live, Razorpay pending).

The scoped MVP introduces a `deliveryTasks` Convex table, automatic delivery task creation at order placement, an admin workflow for assigning and updating delivery status, a customer-facing delivery status view, and failed delivery reason capture. It explicitly does not touch payment, stock, or any external delivery API.

---

## Business Case

### Problem Statement

Every Nuemart order requires physical delivery but has no system support for it. The admin tracks delivery via WhatsApp and phone calls. The customer has no post-order visibility beyond the order status. There is no audit trail for disputes, failed deliveries, or delivery history. This gap affects every order placed and grows more costly as order volume increases.

### Proposed Value

- **Admin:** Structured delivery workflow embedded in existing admin order view. Full audit trail per order. No external tool dependency.
- **Customer:** Delivery status visibility on their Order Detail page — reduces support contacts ("Where is my order?").
- **Business:** Operational data captured from day one. Enables future delivery analytics, rider assignment, and proof of delivery once volume warrants it.

## Scope

### What is in scope

- `deliveryTasks` Convex table — linked to orderId, userId, addressId
- Automatic delivery task creation (status = Pending) as part of order placement mutation
- Admin: assign delivery person name + contact inside Admin Order Detail (SCR-ADM-0009)
- Admin: update delivery status through 7-state lifecycle (Pending → Assigned → Picked Up → Out for Delivery → Delivered / Failed / Cancelled)
- Admin: capture failed delivery reason via fixed 8-option dropdown + optional notes (mandatory when "Other" selected)
- Customer: read-only delivery status + delivery address on Order Detail (SCR-CUS-0010). No delivery person name shown.
- Three hard independence rules: delivery updates do NOT affect order status, payment status, or stock

### What is explicitly out of scope

- Standalone `/admin/delivery` screen (deferred)
- Delivery person login or mobile interface
- Automated customer notifications (SMS, WhatsApp, email)
- Proof of delivery (OTP, photo, signature)
- Third-party delivery partner API integration (Dunzo, Shadowfax, Porter)
- Live map tracking
- Delivery fee calculation
- Delivery SLA monitoring and reporting
- Failed delivery retry / reassignment workflows
- Pickup / self-collection fulfilment type

## Options Considered

### Option A — Build Delivery MVP Now (Recommended)

Build the scoped MVP as confirmed in GRILLING-0001: automatic delivery task creation at order placement, admin assignment and status management, customer status visibility. No external dependencies. Embedded in existing screens.

**Pros:**
- Fills the delivery visibility gap immediately
- Captures delivery data from the first real order
- Low complexity, low risk — no external APIs or new roles
- Independent of Razorpay — can be built in parallel with or before Phase 11
- Schema designed to support future model expansion without rework
- Foundation for all future delivery capabilities (proof of delivery, rider app, notifications)

**Cons:**
- Adds a new Convex mutation to the order placement flow — requires careful testing
- Admin must manually update all delivery statuses — no automation

---

### Option B — Defer Until After Razorpay (Phase 11)

Do not build the delivery module until Razorpay payment integration is live. Continue manual delivery tracking via phone and WhatsApp in the interim.

**Pros:**
- Keeps current development focus on revenue-critical Razorpay integration
- Avoids adding any changes to the order placement mutation before payments are live

**Cons:**
- Every order placed without the delivery module is an order with no system audit trail
- Manual tracking debt grows with order volume
- Customer experience gap persists
- Delivery module build is straightforward enough that it does not compete with Razorpay for development attention

---

### Option C — Do Not Build

Keep delivery entirely manual indefinitely. Accept that delivery is an operational rather than a product concern.

**Pros:**
- Zero development effort

**Cons:**
- No audit trail — disputes cannot be resolved with data
- Customer experience: no post-order visibility beyond order status
- Operational scaling: manual tracking breaks down above ~20–30 orders per day
- Future delivery capabilities (proof of delivery, rider app, partner integrations) cannot be built without this foundation
- Module is already scoped in grilling — cost of inaction is concrete

---

## Recommendation

**Recommended option:** Option A — Build Delivery MVP Now

**Reasoning:** The MVP scope is well-bounded, complexity is low (score: 2), risk is low (score: 2), and business value is meaningful (score: 3). There are no external service dependencies — no third-party APIs, no new user roles, no payment integration required. The module is independent of Razorpay and can be built in any order relative to Phase 11. Every order placed without the delivery module is a missed opportunity to capture operational data. Building it now sets the data foundation for all future delivery capabilities.

**Conditions / caveats:**
- Delivery task creation must be part of the order placement mutation (or a guaranteed compensating action). Silent delivery task creation failures are not acceptable.
- Status transition rules (BR-13 in GRILLING-0001) must be enforced server-side, not only in the UI.
- Failed reason enum values should be defined in the Convex schema (not as raw strings) to support future filtering and reporting.

## User Story

```
As an admin,
I want to cancel a delivery task with an optional reason,
So that I can record that the delivery will not proceed without affecting the order, payment, or stock.
```

---

## Acceptance Criteria

- [ ] **AC-1 (Cancel option available):**
  Given a delivery task exists and status is not "delivered", "failed", or "cancelled",
  When admin views the Delivery section,
  Then a "Cancel Delivery" option is available.

- [ ] **AC-2 (Happy path — cancel without reason):**
  Given admin clicks "Cancel Delivery" and confirms without entering a reason,
  When the mutation completes,
  Then `status = "cancelled"`, `cancelledAt` is set, `updatedAt` and `updatedBy` recorded. `cancelledReason` is null.

- [ ] **AC-3 (Cancel with optional reason):**
  Given admin enters "Customer requested cancellation" as the reason and confirms,
  When the mutation completes,
  Then `status = "cancelled"`, `cancelledAt` set, `cancelledReason = "Customer requested cancellation"`.

- [ ] **AC-4 (Cancel from any non-terminal status):**
  Given delivery status is "pending", "assigned", "picked_up", or "out_for_delivery",
  When admin cancels,
  Then the cancellation succeeds regardless of the current status.

- [ ] **AC-5 (Cannot cancel from terminal status):**
  Given delivery status is "delivered" or "failed",
  When the `cancelDelivery` mutation is called,
  Then it is rejected with error: "Cannot cancel a delivery that is already [status]."

- [ ] **AC-6 (Three independence rules):**
  Given admin cancels delivery,
  Then order status, payment status, and stock are all unchanged.

- [ ] **AC-7 (Section read-only after cancel):**
  Given status = "cancelled",
  When admin views Delivery section,
  Then section is read-only with cancelled status badge and reason (if provided).

- [ ] **AC-8 (Admin auth enforced):**
  Given a non-admin caller,
  When `cancelDelivery` mutation is called,
  Then it throws `ConvexError("Unauthorized")`.

## Business Rules

| # | Rule |
|---|---|
| BR-1 | `cancelledReason` is optional — cancel is valid with or without a reason |
| BR-2 | Cancellation is valid from: pending, assigned, picked_up, out_for_delivery |
| BR-3 | Cancellation is NOT valid from: delivered, failed (terminal states) |
| BR-4 | Delivery cancellation must NOT change order status, payment status, or stock |
| BR-5 | `cancelledAt` set by server — not accepted from client |
| BR-6 | `updatedBy` resolved from `ctx.auth` |

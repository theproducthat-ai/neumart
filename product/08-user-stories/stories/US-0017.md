## User Story

```
As an admin,
I want to view and update allergen and ingredient details when editing an existing product,
So that allergen information can be added and corrected progressively over time.
```

---

## Acceptance Criteria

- [ ] **AC-1 (Fields pre-populated):**
  Given a product has existing allergen data,
  When an admin opens `/admin/products/[id]/edit`,
  Then the allergen fields are pre-populated with the product's current allergen values

- [ ] **AC-2 (Fields blank for products with no data):**
  Given a product has no allergen data,
  When an admin opens the edit form,
  Then all allergen fields are blank (not errored or disabled)

- [ ] **AC-3 (Update allergen data):**
  Given a product has `containsAllergens: ["Milk"]`,
  When an admin changes it to `["Milk", "Soy"]` and saves,
  Then `containsAllergens` is updated to `["Milk", "Soy"]` and all other product fields are unchanged

- [ ] **AC-4 (Clear allergen data):**
  Given a product has `containsAllergens` set,
  When an admin deselects all values and saves,
  Then `containsAllergens` is saved as undefined (or empty); product detail page shows fallback message

- [ ] **AC-5 (Add allergen data to previously unenriched product):**
  Given a product has no allergen data,
  When an admin adds allergen information and saves,
  Then allergen data is persisted and visible to customers immediately

## Business Rules

| # | Rule |
|---|---|
| BR-1 | Editing allergen fields must not affect any other product fields (name, price, stock, etc.) |
| BR-2 | Multi-select values for `containsAllergens` and `mayContainAllergens` are from the fixed 12-item allergen list only |
| BR-3 | Multi-select values for `dietaryTags` are from the fixed 10-item dietary tag list only |
| BR-4 | Clearing all allergen fields and saving is valid — no validation error |
| BR-5 | Pre-population must reflect the currently saved database values, not a cached or stale version |

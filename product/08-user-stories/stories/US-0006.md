## User Story

```
As an admin,
I want to mark a delivery as failed and record a reason from a fixed list,
So that there is an audit record of why the delivery did not succeed, without affecting the order, payment, or stock.
```

---

## Acceptance Criteria

- [ ] **AC-1 (Mark as Failed option available):**
  Given a delivery task has status = "out_for_delivery",
  When admin views the Delivery section,
  Then a "Mark as Failed" button or option is visible alongside "Mark as Delivered".

- [ ] **AC-2 (Failed form shown on click):**
  Given admin clicks "Mark as Failed",
  When the form appears,
  Then a required dropdown of 8 reason options is shown, plus an optional notes textarea.

- [ ] **AC-3 (Happy path — reason selected):**
  Given admin selects "Customer unavailable" from the dropdown and clicks confirm,
  When the mutation completes,
  Then `status = "failed"`, `failedReasonCode = "customer_unavailable"`, `failedAt` is set, `updatedAt` and `updatedBy` are recorded. `failedReasonNotes` is null.

- [ ] **AC-4 (Reason required):**
  Given admin clicks confirm without selecting a reason,
  Then form submission is blocked with error: "Please select a failed delivery reason."

- [ ] **AC-5 (Notes required when "Other"):**
  Given admin selects "Other" from the dropdown and leaves notes blank,
  Then form submission is blocked with error: "Please add notes when selecting Other."

- [ ] **AC-6 (Notes saved when "Other" with text):**
  Given admin selects "Other" and enters notes "Customer gate was locked",
  When the mutation completes,
  Then `failedReasonCode = "other"`, `failedReasonNotes = "Customer gate was locked"`, `failedAt` set.

- [ ] **AC-7 (Notes optional for other reasons):**
  Given admin selects "Wrong address" and does NOT enter notes,
  When admin confirms,
  Then the mutation succeeds. `failedReasonNotes` is null.

- [ ] **AC-8 (Three independence rules):**
  Given admin marks delivery as failed,
  Then `orders.status` is unchanged, `payments.status` is unchanged, `products.stock` is unchanged.

- [ ] **AC-9 (Section read-only after Failed):**
  Given status = "failed",
  When admin views Delivery section,
  Then reason and notes are displayed. No further status update is possible. Section is read-only.

## Business Rules

| # | Rule |
|---|---|
| BR-1 | Failed delivery can only be set from "out_for_delivery" status (valid transition per BR-6 in PRD) |
| BR-2 | `failedReasonCode` is required when status = "failed" |
| BR-3 | `failedReasonNotes` is required when `failedReasonCode = "other"`, optional for all others |
| BR-4 | Failed delivery must NOT change order status, payment status, or stock |
| BR-5 | `failedAt` is set by the server at the time of the mutation |
| BR-6 | `updatedBy` resolved from `ctx.auth` — not from client argument |

---

## Validation Rules

| Field | Rule | Error Message |
|---|---|---|
| `failedReasonCode` | Required. Must be one of 8 valid enum values. | "Please select a failed delivery reason." |
| `failedReasonNotes` | Required when `failedReasonCode = "other"`. Optional otherwise. | "Please add notes when selecting Other." |
| Status transition | Must be "out_for_delivery" to transition to "failed" | "Cannot set delivery to 'failed' from current status '[current]'." |
| Admin role | Verified server-side | `ConvexError("Unauthorized")` |

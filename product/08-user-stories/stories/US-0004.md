## User Story

```
As an admin,
I want to assign a delivery person (name and optional contact) to a delivery task,
So that there is a named person responsible for the delivery and the task transitions from Pending to Assigned.
```

---

## Acceptance Criteria

- [ ] **AC-1 (Assign form visible when Pending):**
  Given a delivery task has status = "pending",
  When admin views the Delivery section on Admin Order Detail,
  Then an assignment form is visible with a text input for delivery person name and an optional text input for contact number.

- [ ] **AC-2 (Happy path — assigns successfully):**
  Given admin enters a delivery person name and clicks "Assign Delivery",
  When the mutation completes,
  Then `status` updates to "assigned", `assignedTo` is saved, `assignedAt` is set to current timestamp, `updatedAt` is updated, and `updatedBy` is set to the admin's user ID.

- [ ] **AC-3 (Contact is optional):**
  Given admin enters only a delivery person name (no contact),
  When they click "Assign Delivery",
  Then the assignment succeeds. `assignedContact` is null/undefined.

- [ ] **AC-4 (Name is required):**
  Given admin clicks "Assign Delivery" without entering a name,
  Then form validation blocks submission with error: "Delivery person name is required to assign."

- [ ] **AC-5 (Assign form hidden when not Pending):**
  Given a delivery task has status = "assigned" or later,
  When admin views the Delivery section,
  Then no assign form is shown — the section shows assigned person details and the next action controls.

- [ ] **AC-6 (Admin auth enforced server-side):**
  Given a non-admin user,
  When they call `assignDelivery` mutation directly,
  Then the mutation throws `ConvexError("Unauthorized")`.

## Business Rules

| # | Rule |
|---|---|
| BR-1 | `assignedTo` (delivery person name) is required to assign. `assignedContact` is optional. |
| BR-2 | Assigning is only valid when current status is "pending". Server-side check required. |
| BR-3 | `assignedAt` is set by the server — not accepted from client. |
| BR-4 | `updatedBy` is resolved from `ctx.auth` — not accepted from client (DEC-003). |
| BR-5 | Assignment does not affect order status, payment status, or stock (BR-1 through BR-3 from PRD). |
| BR-6 | Delivery person is stored as inline strings only — no FK to a `deliveryPersons` table (DEC-014). |

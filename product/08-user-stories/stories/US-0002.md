## User Story

```
As the system,
I want to automatically create a delivery task when a customer places an order,
So that every order has a delivery record from the moment it is placed, with no admin action required.
```

---

## Acceptance Criteria

- [ ] **AC-1 (Happy path — task created):**
  Given a customer has items in cart and a saved address,
  When they complete the checkout flow and the order is placed successfully,
  Then a `deliveryTasks` record is created with `status = "pending"`, `orderId` matching the new order, `userId` matching the customer, and `addressId` matching the selected checkout address.

- [ ] **AC-2 (Atomicity — failure rolls back):**
  Given the order placement mutation is called,
  When delivery task creation fails (e.g. schema error or database error),
  Then the order itself is NOT created — no orphaned order exists without a delivery task.

- [ ] **AC-3 (createdAt is set):**
  Given a delivery task is created at order placement,
  When the record is inspected,
  Then `createdAt` is set to a valid Unix timestamp close to the order's own `createdAt`.

- [ ] **AC-4 (updatedAt and updatedBy):**
  Given a delivery task is created at order placement,
  When the record is inspected,
  Then `updatedAt` equals `createdAt` and `updatedBy` is set to the placing customer's userId (system-initiated, not admin).

- [ ] **AC-5 (One task per order):**
  Given an order is placed,
  When checked in the database,
  Then exactly one `deliveryTasks` record exists for that `orderId`.

- [ ] **AC-6 (No stock or payment change):**
  Given the delivery task is created at order placement,
  Then no additional stock deduction occurs beyond the existing order placement logic, and no payment record is created or modified by the delivery task creation.

## Business Rules

| # | Rule |
|---|---|
| BR-1 | Delivery task creation is part of the same Convex mutation as order placement — not a separate async call (DEC-012) |
| BR-2 | If delivery task creation fails, the order mutation must also fail — no partial state |
| BR-3 | Default `status` on creation is always `"pending"` |
| BR-4 | `addressId` stored on `deliveryTasks` is the address the customer selected at checkout |
| BR-5 | Delivery task creation must NOT trigger any additional stock deduction or payment creation |

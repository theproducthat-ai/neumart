## User Story

```
As the system,
I want to automatically generate a unique customerCode and qrCodeId
for every Nuemart customer,
so that each customer has a permanent, public-safe identity token
that can be used in QR codes without exposing internal IDs.
```

---

## Acceptance Criteria

- [ ] **AC-1 (New user — auto-generation):**
  Given a new user completes sign-up via Clerk
  When the `syncUser` mutation runs
  Then the user's Convex record contains a unique `customerCode` (format: `CUST-NNNNNN`) and a unique `qrCodeId` (min 10 chars, URL-safe)

- [ ] **AC-2 (Existing user — lazy generation, first visit):**
  Given an existing user has no `qrCodeId` on their Convex record
  When the `generateQrCode` mutation is called for that user
  Then `customerCode`, `qrCodeId`, `qrEnabled: true`, and `qrCreatedAt` are written to their record

- [ ] **AC-3 (Idempotency):**
  Given a user already has `qrCodeId` and `customerCode` set
  When `generateQrCode` is called again for the same user
  Then no changes are made — the existing values are preserved exactly

- [ ] **AC-4 (Uniqueness — qrCodeId):**
  Given two different users are created
  Then each user has a different `qrCodeId` — no two users share the same value

- [ ] **AC-5 (Uniqueness — customerCode):**
  Given two different users are created
  Then each user has a different `customerCode` — no two users share the same value

- [ ] **AC-6 (qrCodeId safety):**
  Given a user's `qrCodeId` is inspected
  Then it is not equal to their Convex `_id` or their Clerk `tokenIdentifier`

- [ ] **AC-7 (Index — lookup by qrCodeId):**
  Given the `by_qr_code_id` index exists on the `users` table
  When a query looks up a user by `qrCodeId`
  Then the query resolves without a full table scan

## Business Rules

| # | Rule |
|---|---|
| BR-1 | `qrCodeId` must be globally unique across all users (PRD BR-1) |
| BR-2 | `customerCode` must be globally unique; format: `CUST-` + 6-digit zero-padded sequence (PRD BR-2) |
| BR-3 | `qrCodeId` must never equal the Convex `_id` or Clerk `tokenIdentifier` (PRD BR-3) |
| BR-4 | Once set, `qrCodeId` and `customerCode` must never be overwritten (PRD BR-4) |
| BR-7 | `generateQrCode` is idempotent — second call makes no changes (PRD BR-7) |

---

## Validation Rules

| Field | Rule | Error Message |
|---|---|---|
| `qrCodeId` (generation) | Must be unique — check before insert | Retry with new nanoid on collision (log collision — extremely rare) |
| `customerCode` (generation) | Must be unique — derived from sequence or checked before insert | Retry with next sequence value on collision |

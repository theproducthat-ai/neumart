## User Story

```
As an admin,
I want to update the delivery status step by step through the delivery lifecycle (Picked Up → Out for Delivery → Delivered),
So that I can track and record the progress of every delivery with accurate timestamps.
```

---

## Acceptance Criteria

- [ ] **AC-1 (Next status button shown):**
  Given a delivery task is in status "assigned",
  When admin views the Delivery section,
  Then a "Mark as Picked Up" button is visible.

- [ ] **AC-2 (Happy path — Picked Up):**
  Given status is "assigned",
  When admin clicks "Mark as Picked Up",
  Then `status` updates to "picked_up", `pickedUpAt` is set, `updatedAt` and `updatedBy` are recorded.

- [ ] **AC-3 (Happy path — Out for Delivery):**
  Given status is "picked_up",
  When admin clicks "Mark as Out for Delivery",
  Then `status` updates to "out_for_delivery", `outForDeliveryAt` is set, `updatedAt` and `updatedBy` are recorded.

- [ ] **AC-4 (Happy path — Delivered):**
  Given status is "out_for_delivery",
  When admin clicks "Mark as Delivered",
  Then `status` updates to "delivered", `deliveredAt` is set, `updatedAt` and `updatedBy` are recorded.

- [ ] **AC-5 (No further updates when Delivered):**
  Given status is "delivered",
  When admin views the Delivery section,
  Then no status update button is shown. Section is read-only.

- [ ] **AC-6 (Invalid transition rejected server-side):**
  Given status is "assigned",
  When a client attempts to transition directly to "delivered" (bypassing intermediary states),
  Then the server mutation rejects the transition with error: "Cannot set delivery to 'delivered' from current status 'assigned'."

- [ ] **AC-7 (Independence — order status unchanged):**
  Given admin marks delivery as "delivered",
  Then order status, payment status, and stock are all unchanged.

- [ ] **AC-8 (Admin auth enforced):**
  Given a non-admin caller,
  When `updateDeliveryStatus` mutation is called,
  Then it throws `ConvexError("Unauthorized")`.

## Business Rules

| # | Rule |
|---|---|
| BR-1 | Valid transitions for this story: assigned → picked_up → out_for_delivery → delivered |
| BR-2 | Each transition records its specific timestamp field (`pickedUpAt`, `outForDeliveryAt`, `deliveredAt`) |
| BR-3 | `updatedAt` is refreshed on every status change |
| BR-4 | `updatedBy` is resolved from `ctx.auth` — not from client argument |
| BR-5 | Status = "delivered" is a terminal state for successful delivery — no further updates |
| BR-6 | Delivery status updates must NOT write to `orders`, `payments`, or `products` tables |

---

## Validation Rules

| Field | Rule | Error Message |
|---|---|---|
| Status transition | Must follow valid sequence | "Cannot set delivery to '[new]' from current status '[current]'." |
| Admin role | Verified server-side | `ConvexError("Unauthorized")` |
| Current status | Must be correct predecessor for the requested transition | Same error as above |
